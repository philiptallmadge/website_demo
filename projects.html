  • Distributed File System

    This project was completed for my distributed systems class and explored the tradeoffs between consistency, availability, and partition tolerance, better known as the CAP theorem.

    System Architecture
    This system had a central "name server" which was responsible for handling all initial user requests. It then had a preallocated number of "storage nodes", which could be changed on server boot. The name server was responsible for maintaining an internal hashing of file names to file ids, which were integers. On file creation/deletion, this name server would make calls to storage nodes to create/delete files in their directories. These files were known only by their file id to the storage nodes, which allowed us to have faster rename times, as only the name server had to update its database. On read/writes, the client first sends a request to the name server with a filename. The server then checks its database to convert the filename to a file id, then from a file id to a storage node address. The name server returns this address to the client. The client then can send requests to the storage node. The client system was written such that clients must connect to the name server first, even if they previously have accessed a given file. This ensures they are reading the most updated information possible, as the file may have been deleted between requests.

    Further Information
    • This codebase was entirely written in Python and could be made faster by converting to a lower level language, such as C++
    • As a result of using Python, our distributed file system was slower than the traditional Unix file system
    • The name server does not create storage nodes, but instead listens for them on boot to update its address database. This could be improved by having storage nodes send heartbeats to the name server to ensure they are still active.
